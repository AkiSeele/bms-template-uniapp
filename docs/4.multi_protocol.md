# 4. 多协议热兼容与策略模式

为了无缝兼容多厂商、不同报文格式的 BMS 电池硬件，项目严格遵循开闭原则（OCP），在数据层与视图层采用了**“双注册表自适应架构”**。

---

## 一、 双注册表自适应架构

双注册表相互独立但通过 `protocolType`（唯一小写连字符标识）进行逻辑挂接：

```
                    ┌────────────────────────┐
                    │      GATT 蓝牙连接     │
                    └───────────┬────────────┘
                                │ (获取主服务 UUID)
                                ▼
         ┌──────────────────────────────────────────────┐
         │ 数据层注册表: service/protocol/protocol-registry │
         │   依据服务 UUID, 匹配并实例化对应的解析策略    │
         └──────────────────────┬───────────────────────┘
                                │ (activeProtocolParser.protocolType)
                                ▼
         ┌──────────────────────────────────────────────┐
         │ 视图层注册表: components/panel-registry.ts   │
         │   依据协议标识符, 静态路由返回对应的 UI 组件  │
         └──────────────────────┬───────────────────────┘
                                │
                                ▼
                       [ 渲染对应品牌控制面板 ]
```

---

## 二、 协议解析策略契约

所有新建的厂商协议解析器，必须实现 [types/protocol.ts](../types/protocol.ts) 中声明的 `BmsProtocolParser` 接口。核心属性：
*   `protocolType`: 唯一小写连字符标识（如 `'protocol-a'`）。
*   `protocolName`: 友好显示名称。
*   `spec`: 声明协议规格说明 `ProtocolSpec` 常量（包含帧头、小端/大端字节序、校验算法、校验范围、指令集字典）。
*   `pollingCycleLength`: 轮询指令的循环周转周期长度（步数），指示 Store 需要在此周转步数内拉取全部遥测指令，以此实现 Store 与具体协议的周期彻底解耦。
*   `getPollCommands(step: number)`: 定时轮询生成器（支持多帧循环轮转）。
*   `parseReceivedData(bytes: Uint8Array)`: 执行拼包帧的解码验证，将其解析映射为标准的 `BmsTelemetryUpdate` 增量数据包。
*   `getControlCommand(type, open)`: 执行充电开关、放电开关、加热控制等指令的组包。

---

## 三、 动态接入新协议三步法

> [!IMPORTANT]
> **命名硬对齐规范红线（强约束）**：
> 在 `config/index.ts` 中定义的 `BLE_SERVICES` 配置键名（Key），将被框架直接提取并作为该协议的唯一 `protocolType` 标识符。它**必须与数据解析策略文件名、专属 UI 面板组件名保持百分之百的一致**：
> 1. 数据层策略文件名必须为：`service/protocol/${Key}.ts`
> 2. 视图层专属面板文件名必须为：`pages/*/components/${Key}.vue`
> 
> **一致性校验与报错引导机制**：
> *   **TS 数据层**：若配置了键名但未创建对应的 `service/protocol/${Key}.ts`，[protocol-registry.ts](../service/protocol/protocol-registry.ts) 将在编译期/运行期控制台抛出明确的 `console.error` 指引。
> *   **Vue 视图层**：若未在对应的 `pages/index/components/`、`pages/control/components/`、`pages/param/components/` 下创建 `${Key}.vue` 组件，[panel-registry.ts](../components/panel-registry.ts) 将在控制台打印 `console.warn` 警告并引导降级渲染 `default` 面板。

假定你需要接入硬件协议 `Protocol C`（命名标识符统一为小写连字符 `protocol-c`）：

### 第一步：在 `config/index.ts` 中配置 UUID 服务集
在 `BLE_SERVICES` 中新增以 `protocol-c` 作为键名的配置项：
```typescript
// config/index.ts
"protocol-c": [
  {
    serviceId: "0000FFF0-0000-1000-8000-00805F9B34FB",
    writeCharacteristicId: "0000FFF1-0000-1000-8000-00805F9B34FB",
    notifyCharacteristicId: "0000FFF2-0000-1000-8000-00805F9B34FB",
  }
] as BleServiceConfig[],
```

### 第二步：在 `service/protocol/` 目录下创建协议策略文件
新建 `protocol-c.ts` 文件，编写组包解包逻辑：
```typescript
import { BmsProtocolParser, BmsTelemetryUpdate, ProtocolSpec } from "@/types/protocol";

const PROTOCOL_SPEC: ProtocolSpec = {
  frameHeader: [0x5A],
  minFrameLength: 6,
  byteOrder: "big-endian",
  checksumAlgorithm: "crc16-modbus",
  checksumRange: { start: 0, endOffset: 2 },
  commandSet: {
    READ_INFO: "5A 03 00 00 00 0A C5 CD",
    CTRL_CHARGE_ON: "5A 06 01 00 00 01 C5 CD"
  }
};

export class ProtocolCParser implements BmsProtocolParser {
  readonly protocolType = "protocol-c";
  readonly protocolName = "Protocol C (Modbus RTU)";
  readonly spec = PROTOCOL_SPEC;
  readonly pollingCycleLength = 1;

  getPollCommands(step: number): string[] {
    return [PROTOCOL_SPEC.commandSet["READ_INFO"]!];
  }

  parseReceivedData(bytes: Uint8Array): BmsTelemetryUpdate | null {
    // 基础校验与解码
    return {
      totalVoltage: 52.4,
      realtimeCurrent: 10.2,
      batteryPercent: 88,
      extendedData: { soh: 99 }
    };
  }

  getControlCommand(type: string, open: boolean): string {
    return PROTOCOL_SPEC.commandSet["CTRL_CHARGE_ON"]!;
  }
}
```

### 第三步：编译期自动检测与物理注册

本项目在 [service/protocol/protocol-registry.ts](../service/protocol/protocol-registry.ts) 中，已经基于 Vite 编译期 Glob 静态扫描所有策略解析器文件并自动关联配置完成注册：
```typescript
// 自动加载该目录下除 registry 外的所有 ts 协议解析类
const protocolModules = import.meta.glob("./!(protocol-registry).ts", { eager: true });
```
这意味着：
1. 开发者**不需要**手动修改注册表文件代码。
2. 系统会提取协议文件的文件名（如 `protocol-c.ts`）作为协议标识符 `protocolType`。
3. 它会自动将文件名与 `config/index.ts` 中 `BLE_SERVICES` 下的配置键名（例如配置为 `protocol-c`）进行自适应物理注册。

## 四、 微信小程序下 UI 面板自适应与自动装载

*   **平台限制**: 微信小程序不支持运行时动态 `import()` 组件，并且直接在模板中以组件字符串名称动态渲染 `<component :is="'component-name'">` 在小程序编译时会报错中断构建。
*   **自适应 Glob 自动注册**:
    项目在 [components/panel-registry.ts](../components/panel-registry.ts) 中，通过 Vite 编译期 Glob 静态扫描，查找各个业务页面下的专属面板组件，并以文件名作为 `protocolType` 键进行注册：
    ```typescript
    const homeModules = import.meta.glob("../pages/index/components/*.vue", { eager: true });
    ```
    因此，新增专属面板时，只需在页面组件目录中创建对应的 `protocolType.vue` 组件即可（例如 `pages/index/components/protocol-c.vue`），它会自动被注册表扫描到并映射，**无需任何代码侵入与手动 import**！
*   **页面渲染分流机制**:
    在需要支持小程序的页面容器中，必须使用条件编译进行分支分流：
    - **微信小程序端 (`#ifdef MP-WEIXIN`)**: 必须通过 `v-if` / `v-else-if` 静态导入并分流渲染。
    - **非微信小程序端 (`#ifndef MP-WEIXIN`)**: 允许自适应解析组件对象引用并使用 `<component :is>` 动态挂载。

以主页 `pages/index/index.vue` 为例：
```vue
<!-- #ifdef MP-WEIXIN -->
<view v-show="activeTab === 'realtime'">
  <realtime-default-panel v-if="currentProtocolType === 'default'" />
</view>
<!-- #endif -->

<!-- #ifndef MP-WEIXIN -->
<view v-show="activeTab === 'realtime'">
  <component :is="homePanelComponent" v-if="homePanelComponent" />
</view>
<!-- #endif -->
```
此方案完美绕过了小程序编译机制的硬限制，在各端均能实现极佳的 UI 渲染兼容性。
