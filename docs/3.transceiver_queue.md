# 3. 指令收发与队列调度机制

由于低功耗蓝牙通信是单线程串行的（GATT 协议只支持单事务一问一答），如果在多端高频操作下不加控制，极易因为“轮询指令”与“控制指令”在空口碰撞，造成硬件解析失败、丢包或超时阻断。

为此，[ble-store.ts](../stores/ble-store.ts) 设计了一套**“单一通道指令优先级排队调度队列”**。

---

## 一、 队列调度时序

```
                       [ 控制指令 / 轮询指令 ]
                               │
                               ▼
        ┌──────────────────────────────────────────────┐
        │                 指令队列调度                 │
        │  * 控制指令: 清空队列普通轮询, 并插队最前方  │
        │  * 轮询指令: 去重防堆积, 放入队列尾部        │
        └──────────────────────┬───────────────────────┘
                               │
                               ▼
                        [ 物理队列 (Task Queue) ]
                               │
                               ├─► 执行当前任务 ──► 物理分包下发 (50ms 延时)
                               │      ▲
                               │      │ (超时/未响应重试)
                               ▼      │
                        [ 超时定时器: 1500~2500ms ]
                               │
                ┌──────────────┴──────────────┐
                │                             │ (成功接收响应帧)
                ▼                             ▼
           [ 宣告超时 ]                  [ 清除定时器 ]
          重试次数用尽时               执行 Promise.resolve
         静默/报错推进下个任务           推进并处理下个任务
```

---

## 二、 指令调度细则

### 1. 指令优先级划分 (Priority Rules)
*   **控制指令 (手动物理控制，如 charge/discharge)**:
    享有极高优先级。控制任务入队前，会**清空队列中所有尚未执行的普通轮询遥测任务**，防止慢速遥测查询占用有限信道。随后，将控制指令**直接插入队列最前端**（插队），强行立刻执行。
*   **轮询指令 (定时读取电池遥测数据)**:
    在推入队列前进行去重防堆积处理：若队列中已有相同命令字的任务，或当前已存在控制任务（或处于控制响应等待状态），则**直接略过，丢弃本次轮询任务**，以防止指令堆积导致通道阻塞。

### 2. 指令动态分包下发 (Chunking)
在底层的 `writeCommand` 中：
*   **MTU 动态分包**: 根据当前协商成功的 `currentMtu`（默认 20 字节，协商成功后为 `MTU - 3`）对发送包的二进制 ArrayBuffer 进行动态 `slice` 切片。
*   **包间物理延时**: 切片下发时，**包与包之间强制休眠 50 毫秒**（`setTimeout` 挂起），以防止连续写入大包数据导致设备串口缓冲区堵塞发生物理丢包。

### 3. 数据接收粘包重组 (Buffer Stitching)
从 `onCharacteristicValueChange` 上报的数据是零散的字节流段，必须进行拼包还原：
1.  **分包拼接**: 收到数据后将其追加至 Store 内部私有的 `receiveBuffer` 字节数组。
2.  **帧头定位**: 搜寻协议声明的 `frameHeader`（如 `0xAA`）。如果当前缓冲区无帧头，则清空无效垃圾字节；若帧头不在 0 位，则切除并丢弃帧头前的全部垃圾碎片。
3.  **动态长度剪裁**: 提取代表包长度的字节（如第三字节 `bytes[2]` 代表 `LEN`），确定整帧总字节数 `totalFrameLen`。
4.  **截取校验**: 
    *   若缓冲区当前长度小于 `totalFrameLen`，则保留内容并跳出循环，继续挂起等待后续分包到达。
    *   若缓冲区足够，则截取该完整帧，并从拼包缓冲区中将这部分数据切除。
    *   调用 `verifyChecksum` 验证帧，验证通过后执行后续数据更新。

### 4. 响应对账与超时重试 (Timeout & Retry)
*   **响应对账**: 成功解出完整物理帧后，比对帧内的 `cmdId` 与队列中当前正在等待执行的任务 `commandId` 是否一致。一致则表示该下行指令得到了握手响应，清除其超时定时器，执行该任务的 `resolve` 承诺。
*   **超时及重试分流**:
    *   **轮询指令**: 设定 `1.5 秒` 响应超时，重试次数为 `0`。超时熔断后静默放行，直接执行 `resolve(null)` 以防阻碍下一个队列的执行。
    *   **控制指令**: 设定 `2.5 秒` 响应超时，并配置 `2 次` 重试发送。若超时重试全部用尽，则执行 `reject` 向上抛出异常，并由全局错误监听器弹窗提示用户。
*   **发送间隔冷却 (QUEUE_SEND_INTERVAL_MS)**:
    每次指令完成执行（不论是成功接收到回复帧，还是遭遇超时、写入失败等异常）后，队列调度器都会根据 [config/index.ts](../config/index.ts) 中的 `QUEUE_SEND_INTERVAL_MS` 设定的延迟时长挂起等待，冷却完毕后才继续拉取并下发队列中的下一个任务。此字段为 `0` 时代表收到回复即刻发送下一包。
*   **断连清理**: 一旦物理蓝牙连接发生改变导致断连，立刻执行 `clearQueue` 清空并释放队列中所有被阻塞的 Promise，防止引发内存泄漏。
