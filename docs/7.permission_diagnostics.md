# 7. 系统权限诊断与自适应修复

由于低功耗蓝牙 (BLE) 扫描在不同宿主端（特别是 Android 和 iOS 原生 APP 以及微信小程序）对于定位和附近设备的控制逻辑差异巨大，项目设计了高度解耦的复合权限诊断与原生跳转修复服务。

---

## 一、 系统权限诊断项划分

系统将诊断条件抽象为四个必要维度（由 `permissionManager.diagnosePermissions()` 进行物理检测）：

| 诊断项键名 | 代表含义 | 平台处理细节 |
| :--- | :--- | :--- |
| `btHardware` | 手机系统蓝牙物理开关 | 全端通过 API 进行开启状态回读，检测是否未开蓝牙 |
| `gpsHardware` | 手机系统定位 GPS 服务 | Android 及小程序必须开启定位硬件，**iOS 原生 APP 端直接置为 true 忽略** |
| `btPermission` | 应用级附近设备/蓝牙扫描权限 | 检查应用是否获得系统层扫码/连接权限授权 |
| `locPermission` | 应用级获取精确位置权限 | Android 及小程序强校验是否获得定位授权，**iOS 忽略并置为 true** |

---

## 二、 权限状态实时诊断交互

*   **静默诊断**: 诊断页 [permission-check.vue](../pages/mine/permission-check.vue) 在 `onShow` 生命周期，以及通过 `uni.onAppShow()` 注册的 App 回到前台事件中，会自动执行静默诊断，更新红绿指示灯，无需用户手动拉起。
*   **手动诊断**: 点击页面“权限诊断”按钮，伴随 `toast.loading` 的拟真诊断延时反馈（500ms），提升用户对诊断进度的直观信任感。

---

## 三、 跨端自适应原生跳转修复

当用户在诊断页中点击某项未就绪的权限单元格，或在搜索页扫描环境缺失时，修复引擎 `requestSettingOrResolve()` 会根据平台执行原生跳转：

### 1. 微信小程序端
直接调用微信官方 API 打开小程序授权设置面板：
```typescript
uni.openSetting();
```

### 2. 安卓原生 APP 端 (HTML5+ 反射)
通过 HTML5+ 的原生反射桥接，调起原生 `ACTION_APPLICATION_DETAILS_SETTINGS` Intent，直接拉起系统设置里当前 App 的“应用详细资料”页面，让用户直接开启权限或开关：
```typescript
const main = plus.android.runtimeMainActivity();
const Intent = plus.android.importClass("android.content.Intent");
const Settings = plus.android.importClass("android.provider.Settings");
const Uri = plus.android.importClass("android.net.Uri");
const intent = new Intent(Settings.ACTION_APPLICATION_DETAILS_SETTINGS);
const uri = Uri.fromParts("package", main.getPackageName(), null);
intent.setData(uri);
main.startActivity(intent);
```

### 3. 苹果 iOS 原生 APP 端 (HTML5+ 反射)
反射调用 iOS 的 `UIApplication`，打开 `app-settings:` 统一应用配置页链接，实现无缝跳转：
```typescript
const UIApplication = plus.ios.importClass("UIApplication");
const NSURL = plus.ios.importClass("NSURL");
const sharedApplication = UIApplication.sharedApplication();
const settingsURL = NSURL.URLWithString("app-settings:");
if (sharedApplication.canOpenURL(settingsURL)) {
  sharedApplication.openURL(settingsURL);
}
```
跳转后，一旦用户从系统设置返回 App，会自动触发 `onAppShow` 刷新回读，使得诊断灯迅速变绿，完成修复闭环。

## 四、 Android 12+ 附近设备 (Nearby Devices) 权限自适应

在 Android 12（API 级别 31）及更高版本中，Google 调整了低功耗蓝牙的权限模型。以往蓝牙扫描强绑定“定位权限”的行为被废除，转而由专用的“附近设备”权限组进行精细化控制。

为了实现多版本 Android 系统与微信小程序环境的无缝兼容，`permissionManager` 采取了以下策略：

### 1. 动态版本判定与权限路由
在进行 `btPermission`（应用级蓝牙权限）检测时，程序会通过 Android 原生 SDK 版本（`Build.VERSION.SDK_INT`）进行分流判定：
*   **Android 11 及以下（SDK < 31）**: 扫描蓝牙依然高度依赖“定位权限”，附近设备权限物理检测在 App 端直接置为 `true` 快速放行。
*   **Android 12 及以上（SDK >= 31）**: 物理检测强制校验以下两个核心权限，必须同时满足方可通行：
    *   `android.permission.BLUETOOTH_SCAN`：用于启动/执行蓝牙适配器扫描与设备搜索。
    *   `android.permission.BLUETOOTH_CONNECT`：用于执行蓝牙物理握手连接、MTU 协商及读写操作。

```typescript
// 核心逻辑源码片段 (service/permission.ts)
if (sdkVersion >= 31) {
  const hasBtScan = context.checkSelfPermission(APP_CONFIG.ANDROID_PERMISSIONS.BLUETOOTH_SCAN) === 0;
  const hasBtConnect = context.checkSelfPermission(APP_CONFIG.ANDROID_PERMISSIONS.BLUETOOTH_CONNECT) === 0;
  state.btPermission = hasBtScan && hasBtConnect;
} else {
  state.btPermission = true; // 低版本自动置为 true
}
```

### 2. 启动时的“混合直申”机制
为了避免用户在设备搜索页首次点击连接时遭遇频繁弹窗，项目在 `App.vue` 的 `onLaunch` 初始化阶段，通过 `permissionManager.requestAppPermissionsOnLaunch()` 执行了**静默预申请**：
*   App 会同时将 `BLUETOOTH_SCAN`、`BLUETOOTH_CONNECT` 与 `ACCESS_FINE_LOCATION` 推入请求队列。
*   低版本 Android 系统会自动过滤并忽略不支持的附近设备权限；高版本 Android 系统则会合并弹出“允许此应用寻找、连接到您的附近设备吗？”的询问框。
*   由于精细位置权限（`ACCESS_FINE_LOCATION`）在被永久拒绝时，再次触发会导致系统强弹提示，项目在启动预申请时会前置使用 `checkSelfPermission` 预检，若已授权（如“使用时允许”）则跳过申请定位权限，规避 Android 系统的这一顽固 Bug。

### 3. Manifest 覆盖补丁：neverForLocation 策略
在打包 Android 原生 App 时，HBuilderX 默认会自动注入 `BLUETOOTH_SCAN` 权限，但其并不携带 `usesPermissionFlags` 标签。这导致在 Android 12+ 设备上，即使声明了附近设备权限，系统仍然会强行要求定位权限授权才能搜索到蓝牙设备。

为解决此打包机制缺陷，项目在 [AndroidManifest.xml](../nativeResources/android/AndroidManifest.xml) 中放置了自定义清单合并补丁：
```xml
<uses-permission
    android:name="android.permission.BLUETOOTH_SCAN"
    android:usesPermissionFlags="neverForLocation"
    tools:replace="android:usesPermissionFlags" />
```
*   **`neverForLocation` 标记**：明确向 Android 系统声明，App 所有的蓝牙扫描行为**绝不用于推断用户的地理位置**。
*   **`tools:replace="android:usesPermissionFlags"` 属性**：告知 Manifest 合并器，强制使用本补丁中的 `usesPermissionFlags` 标签替换掉 HBuilderX 自动注入的默认同名权限属性，避免属性 duplicate 冲突。
通过此 Manifest 补丁，Android 12+ 用户**完全无需额外授予任何位置权限**，只需授权“附近设备”权限即可流畅扫描并连接 BMS 设备，极大降低了用户在使用定位权限时的心理防备与摩擦感。
