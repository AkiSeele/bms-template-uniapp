# 6. 系统安全授权与激活校验

为了防止核心指令和设置功能在未付费或未授权状态下被随意修改，项目引入了本地设备码离线激活校验以及全局授权锁机制。

---

## 一、 特征设备码生成与获取

设备特征识别码由前端离线生成，确保唯一性且不受卸载重装影响：
1.  用户进入激活页面 ([auth.vue](../pages/mine/auth.vue)) 时，程序自动读取本地缓存中的 `code_dev` 键。
2.  若为空，则调用 `generateRandomDeviceCode()` 产生一个唯一的九位大写特征硬件识别码，写入 Storage 中。
3.  用户点击卡片一键复制硬件识别码。程序写入剪贴板后，**留出 200 毫秒延时回读剪贴板数据**，以检测剪贴板权限是否被系统沙箱阻断。若阻断则引导至系统设置。

---

## 二、 离线激活码校验防重放机制

```
  [ 用户特征码 code_dev ] ──► 管理端根据天数/权限 ──► 产生 8 位激活码 ──► 用户填入
                                                                      │
  ┌───────────────────────────────────────────────────────────────────┘
  ▼
[ 离线校验 ]
  1. decodeAuthCode() 解出激活类型 (永久/限时天数)
  2. 根据当前特征码、时间、类型, 在本地重新 calculateAuthCode() 演算校验码
  3. 比对生成的比对码与填入的激活码是否完全相符
        │
        ├─► [ 失败 ]: 弹出激活失败提示
        │
        └─► [ 成功 ]: 
              1. 计算到期时刻戳, 并更新持久化保存至 userStore.saveAuthInfo()
              2. 强行重置 code_dev 为全新九位随机码, 导致旧激活码立刻报废 (防重放)
```

通过这一物理销毁重放标志位的机制，彻底杜绝了同一激活码在同一设备被多次重复刷用或在其它设备重放激活的漏洞。

---

## 三、 全局控制熔断锁设计

为了在前端形成闭环的权限保护：
1.  Pinia 状态仓 [stores/user.ts](../stores/user.ts) 计算属性 `isAuthorized` 实时计算当前截止时刻戳与本地时钟差：
    ```typescript
    const isAuthorized = computed(() => {
      return authEndTime.value > Date.now();
    });
    ```
2.  在主页和控制页中，充电、放电、加热开关的 `@change` 回调，以及紧急维护的点击回调，都会前置判定 `isAuthorized`：
    ```typescript
    const toggleCharge = async (e: any) => {
      const nextVal = e.detail.value;
      if (!isAuthorized.value) {
        // 瞬间将开关物理反转回之前的状态，防空关视觉错觉
        isCharging.value = !nextVal;
        showAuthRequiredDialog(); // 弹出激活引导弹窗
        return;
      }
      // ... 正常下发流程 ...
    };
    ```
通过这一设计，即使底层服务就绪，前台界面也会对非授权用户形成严密的功能阻断。
